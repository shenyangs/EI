## 4. 测试设计方法
### 4.1 等价类划分
- 注册输入：合法邮箱 / 非法邮箱 / 缺失必填项 / 重复邮箱 / 非法用户类型
- 登录输入：正确凭证 / 错误密码 / 不存在邮箱 / 缺失字段
- AI 模型接口：已授权管理员 / 未授权用户 / 无效模型 ID / 缺失字段
- 安全输入：普通文本 / 包含 script / 包含 iframe / `javascript:` 伪协议 / 组合型 XSS 载荷
- 流式响应：完整 SSE 事件 / 被拆分的 SSE JSON / 结束标记 `[DONE]`

### 4.2 边界值分析
- 速率限制：第 60 次请求允许，第 61 次请求触发 429
- 密码修改：长度 < 6 拒绝，长度 >= 6 允许
- 版本/模型 ID：非法字符串应返回 400
- JSON 请求体：合法 JSON 与非法 JSON 分别覆盖

### 4.3 异常场景
- 未配置 `JWT_SECRET` 的生产登录行为
- Next.js 生产模式多路由之间的状态共享
- `TextDecoder` 缺失时的流式解析兼容性
- `.next/types` 类型文件不稳定时的 `tsc` 执行链路
- 未配置 ESLint 时的交互式阻塞

### 4.4 接口交互验证
- `register -> login -> me -> password -> login` 全链路
- `authMiddleware -> checkPermission -> AI 管理接口` 权限传递链路
- `db -> ai-client -> ai-orchestrator` 模型选择链路
- `stream route -> client stream parser` SSE 解析链路

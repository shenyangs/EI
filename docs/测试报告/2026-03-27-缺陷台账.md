## BUG-001 登录后 Token 无法稳定验证
- 状态：已修复，已回归
- 严重程度：严重
- 优先级：高
- 环境信息：本地 Node/Next.js，认证接口
- 前置条件：未配置 `JWT_SECRET`
- 缺陷描述：`generateToken` 与 `validateToken` 在开发环境分别使用随机临时密钥，导致同一 token 在生成后无法再次验签
- 复现步骤：
  1. 调用登录接口获取 token
  2. 使用该 token 调用 `/api/auth/me`
  3. 观察返回 401
- 预期结果：同一进程内生成的 token 可被稳定验证
- 实际结果：token 随机失效
- 根因分析：`lib/jwt.ts` 每次调用都重新生成随机 fallback secret
- 修复方案：改为稳定开发密钥；生产环境继续强制要求显式配置 `JWT_SECRET`
- 证据：自动化用例 `__tests__/api/auth-routes.test.ts`
- 回归结果：通过

## BUG-009 生产模式下注册后登录失败（路由间内存不共享）
- 状态：已修复，已回归
- 严重程度：阻断
- 优先级：高
- 环境信息：Next.js 生产模式，真实 HTTP 联调
- 前置条件：同一进程内执行 `register -> login`
- 缺陷描述：注册接口写入的内存数据，在登录接口中不可见，导致刚注册用户无法登录
- 复现步骤：
  1. `POST /api/auth/register`
  2. 使用相同邮箱密码调用 `POST /api/auth/login`
  3. 观察返回“邮箱或密码错误”
- 预期结果：刚注册用户可立即登录
- 实际结果：登录失败
- 根因分析：各路由 bundle 拿到的是各自独立的模块内存实例
- 修复方案：将内存存储挂到 `globalThis.__EI_MEMORY_STORE__`，让同一 Node 进程内共享
- 证据：真实 HTTP 烟雾测试（端口 3102）
- 回归结果：通过

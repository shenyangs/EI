# EI用户系统设计方案

## 1. 系统架构

### 1.1 技术栈
- 前端：Next.js 15.5.14 + React 19.0.0
- 后端：Next.js API Routes
- 数据存储：内存存储（模拟数据库）
- 认证：JWT（JSON Web Token）

### 1.2 系统模块
- 用户管理：注册、登录、个人信息管理
- 角色管理：角色定义、角色分配
- 权限管理：权限定义、权限分配

## 2. 用户模型设计

### 2.1 用户类型
基于科研圈子的特点，设计以下用户类型：

| 用户类型 | 描述 |
|---------|------|
| 学生 | 本科生、研究生等在校学生 |
| 讲师 | 高校讲师 |
| 副教授 | 高校副教授 |
| 教授 | 高校教授 |
| 博导 | 博士生导师 |
| 管理员 | 系统管理员 |

### 2.2 用户数据结构

```typescript
interface User {
  id: string;
  username: string;
  email: string;
  password: string; // 加密存储
  fullName: string;
  userType: 'student' | 'lecturer' | 'associate_professor' | 'professor' | 'advisor' | 'admin';
  institution: string; // 所属机构
  department: string; // 所属部门
  createdAt: Date;
  updatedAt: Date;
}
```

## 4. 注册和登录流程

### 4.1 注册流程
1. 用户填写注册信息（用户名、邮箱、密码、姓名、机构、部门、用户类型）
2. 系统验证信息合法性
3. 系统创建用户账号
4. 系统分配默认角色和权限
5. 返回注册成功信息

### 4.2 登录流程
1. 用户输入邮箱和密码
2. 系统验证用户凭证
3. 系统生成JWT token
4. 系统返回token和用户信息
5. 前端存储token用于后续请求

## 5. API接口设计

### 5.1 用户相关接口
- `POST /api/auth/register` - 用户注册
- `POST /api/auth/login` - 用户登录
- `GET /api/auth/me` - 获取当前用户信息
- `PUT /api/auth/me` - 更新用户信息
- `PUT /api/auth/password` - 修改密码

### 5.2 角色相关接口
- `GET /api/roles` - 获取所有角色
- `POST /api/roles` - 创建角色（管理员）
- `PUT /api/roles/:id` - 更新角色（管理员）
- `DELETE /api/roles/:id` - 删除角色（管理员）

### 5.3 权限相关接口
- `GET /api/permissions` - 获取所有权限
- `POST /api/roles/:id/permissions` - 为角色分配权限（管理员）
- `GET /api/roles/:id/permissions` - 获取角色权限（管理员）

## 6. 与现有系统集成

### 6.1 项目管理集成
- 项目创建者和参与者需关联到用户系统
- 项目权限控制基于用户角色

### 6.2 AI模型管理集成
- AI模型配置的创建和管理需基于用户权限
- 不同角色对AI模型的访问权限不同

## 7. 安全性考虑

### 7.1 密码安全
- 密码使用bcrypt等算法加密存储
- 密码强度验证

### 7.2 认证安全
- 使用JWT进行身份认证
- token设置合理的过期时间
- 实现token刷新机制

### 7.3 权限验证
- 所有API请求需验证用户身份和权限
- 实现基于角色的访问控制（RBAC）

## 8. 实现计划

1. 设计并实现用户数据模型
2. 实现注册和登录功能
3. 实现权限系统
4. 集成到现有项目管理和AI模型管理功能
5. 测试系统功能和安全性

## 10. 总结

本设计方案针对科研圈子的特点，设计了一套完整的用户系统，包括用户注册、登录、角色管理和权限控制。系统采用现代前端技术栈，与现有项目管理和AI模型管理功能集成，为科研人员提供安全、高效的用户体验。

# AI 功能缺陷分析与修复报告

## 1. 本次排查范围

本次聚焦项目内所有 AI 相关核心模块，覆盖：

- AI 接口层：`/api/ai/draft`、`/api/ai/think`、`/api/ai/stream`、`/api/ai/test-connection`
- AI 核心能力层：`lib/ai/ai-client.ts`、`lib/streaming-ai.ts`
- AI 前端交互层：写作工作台、方向选择、大纲工作台、改稿建议面板、流式 AI 面板
- AI 管理后台：模型管理、模块配置

### 2.3 异常处理测试

## 4. 缺陷详情

### AI-001 `/api/ai/draft` 协议不一致

复现步骤：

1. 打开章节写作页或全文预览页
2. 点击“AI 生成章节内容”或相关草稿生成按钮
3. 前端调用 `/api/ai/draft`

预期结果：

- 接口返回标准 JSON
- 前端能读取 `ok/content/error`

实际结果：

- 接口返回的是 `text/plain` 流
- 前端执行 `response.json()` 时协议不匹配，真实页面链路不可用

修复方案：

- 将 `/api/ai/draft` 改为标准 JSON 返回
- 保留 fallback 逻辑，但统一返回 `{ ok, content, error, fallback }`

### AI-002 `/api/ai/think` 协议不一致

复现步骤：

1. 新建项目页点击 AI 填充字段
2. 方向页/大纲页调用 `/api/ai/think`
3. 前端执行 `response.json()`

预期结果：

- 返回结构化 JSON
- `content.sections`、`content.metadata` 可直接消费

实际结果：

- 接口返回纯文本流
- 前端无法稳定获得结构化字段

修复方案：

- `/api/ai/think` 改为统一 JSON 响应
- 增加结果标准化逻辑，兼容 `content/data/array` 三类结果结构

### AI-003 流式协议不一致

复现步骤：

1. 打开带 `StreamingAiPanel` 的 AI 面板
2. 发起流式任务
3. 观察面板状态变化

预期结果：

- thinking 阶段推进
- content 分段显示
- 完成后正确回调结果

实际结果：

- 服务端发的是简单 `content` 数据
- 前端却按 `status=progress/completed` 协议解析
- 改稿建议任务 `revision_suggestions` 根本不支持

修复方案：

- 重写 `/api/ai/stream` 事件格式
- 增加 `revision_suggestions` 支持
- 统一输出 `thinking/content/revision/[DONE]`
- 前端流面板补上 `revision` 处理与结果 ref 缓存

### AI-004 改稿建议双请求

复现步骤：

1. 打开改稿建议面板
2. 点击“重新生成建议”

预期结果：

- 只发起一次 AI 生成请求

实际结果：

- 同时调用 `/api/ai/revision` 和 `/api/ai/stream`
- 产生重复耗时与结果不一致风险

修复方案：

- 改稿面板改为单一路径
- 只使用流式生成链路，避免重复调用

### AI-005 Gemini 请求格式错误

复现步骤：

1. 配置 Google/Gemini 模型
2. 执行 AI 编排或模型连通性测试

预期结果：

- 正常请求 Gemini API

实际结果：

- 请求未按 Gemini 规范拼接 `?key=...`
- 仍错误保留 Bearer 头，导致调用不稳定甚至失败

修复方案：

- 在 `lib/ai/ai-client.ts` 和 `/api/ai/test-connection` 中统一 Gemini 请求构造
- 使用 query 参数传 key，并移除不需要的 Bearer 头

### AI-006 AI 后台接口缺少权限保护

复现步骤：

1. 直接访问 AI 模型更新/删除接口、模块配置接口、测试连接接口
2. 不携带有效后台权限

预期结果：

- 返回 401/403，禁止访问

实际结果：

- 存在未受保护或保护不完整的后台 AI 接口

修复方案：

- 为相关接口增加 `authMiddleware + checkPermission`
- 按 `ai:update`、`ai:delete`、`ai:read` 区分权限

### AI-007 数据库模拟实现不准确

复现步骤：

1. 查询 `SELECT * FROM ai_models WHERE provider = ?`
2. 或插入缺少 `subject/keywords` 的项目数据

预期结果：

- provider 查询应准确筛选
- 项目字段应兼容可选项

实际结果：

- provider 查询可能误返回全部模型，影响模型选择
- 项目字段类型过严，测试与业务结构不一致

修复方案：

- 修正 provider 查询分支
- 将项目字段改为可选并兼容不同插入形态

### AI-008 SSE 分片解析不稳定

复现步骤：

1. SSE 消息被网络分片
2. 一个 JSON 事件拆成多个 chunk

预期结果：

- 前端仍能正确拼接并解析

实际结果：

- 原实现按 chunk 直接拆行，遇到分片 JSON 就会解析失败

修复方案：

- 在 `lib/streaming-ai.ts` 中增加缓冲区
- 按 SSE 事件边界 `\n\n` 解析，而不是按单次读取结果直接解析

## 7. 修复前后对比

### 修复前

- 多个 AI 页面接口协议对不上，真实页面功能易失效
- 流式面板协议断裂，状态推进和结果回传不稳定
- Gemini 模型调用存在结构性错误
- 改稿建议会重复请求
- AI 后台接口权限存在缺口

### 修复后

- AI 草稿、AI 思考、流式改稿三条主链路协议统一
- 流式解析可处理分片 SSE
- Gemini 调用格式统一
- 改稿建议只走一条请求链路
- AI 后台敏感接口补齐鉴权

## 8. 未解决问题

以下问题目前不阻塞构建，也不影响本次 AI 主链路修复，但建议后续继续优化：

- 当前本地环境下真实模型调用仍可能因 `self-signed certificate in certificate chain` 回退到默认内容；这属于部署/证书配置问题，不是本次接口协议修复遗留 bug
- 项目中仍有若干 `react-hooks/exhaustive-deps` 警告
- 部分非 AI 组件存在旧代码结构问题，建议单独做前端代码整洁化

# 联网模型与搜索接入方案

## 1. 当前结论

前端原型已经有了，但**真正的联网搜索能力目前还没有接上**。

现在项目里做的是：

- 页面层已经给“联网补证据”“AI 基座”“模型与搜索状态”留出了位置
- 配置层已经预留了 `MiniMax` 作为主模型提供方
- 还没有真正发起模型请求，也没有真正连接搜索工具

## 2. 为什么不能只接一个会写作的模型

你的系统是论文工作台，不是普通聊天工具。

它至少需要两层能力：

1. `主写作模型`
   负责提纲生成、章节写作、改写、总结。

2. `联网搜索工具层`
   负责查会议规则、查文献标题、查案例页面、拉回结构化证据。

推荐把这两层拆开，而不是赌“模型自己就会联网”。

原因：

- 证据来源更可控
- 更方便做引用追溯
- 后续能切换不同模型，不被单一厂商绑死

## 3. 针对 MiniMax 的建议

建议主写作层优先按下面方式接：

- 提供方：`MiniMax`
- 接口方式：`OpenAI 兼容接口`
- 主模型：`MiniMax-M2.5`

这样做的好处是：

- 接口通用，后续切模型更方便
- 你后面如果要做服务端封装，成本更低

补充说明：

- 按当前官方文档，公开列出的 OpenAI 兼容模型包括 `MiniMax-M2.5`、`MiniMax-M2.5-highspeed`、`MiniMax-M2.1`、`MiniMax-M2.1-highspeed` 和 `MiniMax-M2`
- 当前文档里没有看到官方明确列出的 `MiniMax-M2.7`
- 对于中国区环境，官方文档建议使用 `https://api.minimaxi.com/v1`

## 4. 联网搜索推荐做法

推荐顺序：

1. 第一步：先把主写作模型接通
2. 第二步：把联网搜索独立成工具层
3. 第三步：让论文生成流程按需调用搜索，而不是每一步都联网

比较稳妥的调用场景：

- 会议规则识别
- 文献标题核验
- 背景研究补证据
- 案例页抓取与摘要

## 5. 当前代码里的占位

已经新增：

- `.env.example`
- `lib/ai-runtime.ts`

作用：

- 统一读取模型提供方、模型名、基地址和联网开关
- 避免后面把这些配置散落在页面里

## 6. 安全提醒

你刚才发过一次真实密钥。

这类密钥相当于账号凭证，建议你：

1. 立刻去提供方后台旋转或删除旧 key
2. 重新生成一把新 key
3. 只放进本地 `.env.local`
4. 不要再发在聊天消息里

## 7. 下一步怎么做最稳

推荐顺序：

1. 先安装依赖并跑起当前前端
2. 再加一个服务端模型调用入口
3. 再补搜索工具层
4. 最后把“联网补证据”按钮真正接通

一句话总结：

**主模型负责写，搜索工具负责找，论文工作流负责把两者串起来。**

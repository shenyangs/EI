# Atelier EI Stitch 整站页面蓝图

## 1. 这次重构的原则

这次不是在旧页面上“美化”一下，而是把整个产品重新当成一个新的学术工作台来设计。

必须同时满足这 4 条：

1. 不被当前旧版页面结构限制
2. 保留全部核心产品能力和工作流
3. 桌面端和移动端都要有完整体验
4. 产品逻辑必须和页面结构天然匹配，而不是视觉先跑、逻辑后补

一句话定义：

`Atelier EI = 面向非技术用户的 EI 论文研究与写作操作系统`

它不是聊天产品，不是普通管理后台，也不是营销官网。

## 2. Stitch 当前已经跑出来的设计方向

虽然不同页面由 Stitch 给出了几套不同命名的设计系统，但核心方向已经比较稳定，已经能总结成同一条设计语言：

### 统一气质

- 学术、可信、克制、专业
- 像“研究指挥台”或“数字学术编辑台”
- 不做玻璃卡片堆叠，不做常见 SaaS 后台
- 强调留白、秩序、层级，而不是按钮和模块越多越好

### 统一视觉特征

- 主色偏深蓝、青灰、纸张白、档案色
- 标题更有“期刊/编辑部”气质
- 正文与工具信息分层明确
- 不依赖硬边框分割，而依赖背景层级、留白和版面张力
- 大屏强调“决策”和“总览”，移动端强调“聚焦当前任务”

### 统一产品体验原则

- 每一页都必须回答三个问题：
  - 我现在在哪个阶段
  - 我下一步该做什么
  - 系统能怎么帮我推进
- 每一个重要动作都要给出上下文，不让用户在空白页里发懵
- 写作页必须服务于“写论文”，不是做成聊天窗口

## 3. 整体信息架构

整站建议按 7 个核心页面来重构。

### 1. 首页 / 研究指挥台

作用：
- 让用户快速进入系统
- 明确 Atelier EI 是做什么的
- 让用户开始一个新项目，或继续上次的项目

必须有的模块：
- 产品定位区
- 最近项目入口
- 新建论文项目入口
- 论文类型入口（EI 会议 / EI 期刊 / 核心期刊等）
- 系统状态 / AI 可用性
- 5 步研究流程

桌面端建议：
- 更像研究控制台，入口和工作流一起展示
- 左侧是系统级导航，右侧是当前工作入口
- 首页不是宣传页，而是工作开始页

移动端建议：
- 保留“当前推荐动作”优先级
- 减少横向信息，强化卡片式阶段入口
- 让用户可以快速继续上一个项目

### 2. 项目总览 / 决策台

作用：
- 用户打开某个项目后，先判断项目目前状态
- 决定下一步动作

必须有的模块：
- 项目标题和目标会议信息
- 当前阶段
- 进度条或阶段进度
- 下一步推荐动作
- 当前风险提示
- 缺失材料
- 方法建议
- 规则约束（比如会议格式、字数、结构要求）

桌面端建议：
- 做成真正的“决策驾驶舱”
- 信息密度可以更高，但要有主次
- 最重要的是把“下一步”做得非常显眼

移动端建议：
- 以“当前状态 + 下一步”作为单屏中心
- 其余信息折叠为二级内容

### 3. 研究方向页

作用：
- AI 帮用户拆解研究主题
- 给出多个可选方向
- 帮用户决定往哪条路线继续写

必须有的模块：
- 用户原始研究想法
- AI 提炼后的问题定义
- 候选研究方向
- 每个方向的优势、风险、创新点
- 推荐方向
- 用户确认选择入口

设计重点：
- 不是普通列表页，而是“研究选择界面”
- 用户要能看懂差异，而不是被术语淹没

### 4. 论文框架页

作用：
- 确认论文结构
- 逐章梳理内容
- 管理大纲版本

必须有的模块：
- 当前大纲
- 各章节摘要
- 大纲版本历史
- AI 结构优化建议
- 一键采纳或局部修改

桌面端建议：
- 左边章节树，右边章节说明和调整区
- 适合长时间推敲结构

移动端建议：
- 重点看章节顺序和摘要，不做复杂同屏编辑

### 5. 章节写作工作台

作用：
- 这是整个产品最重要的页面
- 用户真正写作、修改、检查都在这里进行

必须有的模块：
- 章节导航
- 当前章节目标
- 当前章节摘要
- 大型正文编辑区
- AI 操作入口
- 质量评分 / 质量提示
- 证据支持 / 参考材料
- 保存、归档、版本入口

桌面端建议：
- 真正的大工作台结构
- 左边章节导航
- 中间正文编辑区
- 右边 AI 支持、质量检查、证据面板
- 顶部是项目状态与关键动作

移动端建议：
- 保留写作主流程，但减少并列面板
- 单次只聚焦一个任务：写、看建议、看评分、切章节
- 更像“章节专注模式”

### 6. 文献与引用中心

作用：
- 管理参考文献、图表、证据材料
- 让论文内容和支撑材料真正对齐

必须有的模块：
- 已引用文献
- 待处理文献
- 引用格式检查
- 图表和材料管理
- 与章节的关联关系

设计重点：
- 不能只是一个附件仓库
- 要让用户看到“哪些内容支撑了哪些章节”

### 7. 全文预览 / 审阅 / 定稿页

作用：
- 在导出前，统一检查全文
- 做最后的审阅和定稿

必须有的模块：
- 全文连续预览
- 章节完整性检查
- 格式检查
- 风险项汇总
- 导出入口
- 定稿归档

桌面端建议：
- 更偏“编辑部审校台”
- 让用户一眼看到还差什么

移动端建议：
- 以快速审阅和导出确认为主

## 4. 双端设计策略

这次不是“桌面缩小成移动”，也不是“移动放大成桌面”，而是：

- 同一套产品逻辑
- 同一套设计语言
- 两套不同的布局方式

### 桌面端

关键词：
- 决策
- 总览
- 并行信息
- 深度写作
- 编辑台

适合承载：
- 项目全局判断
- 复杂结构调整
- 多面板写作
- 风险与证据并行查看

### 移动端

关键词：
- 聚焦
- 当前任务
- 阶段推进
- 轻量修改
- 快速继续

适合承载：
- 继续上一次工作
- 处理单一章节
- 看 AI 建议
- 快速检查状态

## 5. 产品逻辑和页面必须如何对齐

这是这次最重要的一条。

页面不能只是“看起来高级”，必须让产品逻辑更顺。

所以整个产品都应围绕一个统一骨架来组织：

1. 新建项目
2. 分析研究问题
3. 选择研究方向
4. 生成论文框架
5. 逐章写作
6. 文献与证据补强
7. 全文审阅与导出

每个阶段都要有：
- 当前阶段说明
- 为什么现在做这一步
- 用户需要输入什么
- 系统能提供什么帮助
- 做完之后进入哪一步

如果某一页不能清楚说明这 5 件事，这一页就要重做。

## 6. 接下来的实现顺序

代码重构时，建议按这个顺序做：

1. 首页
2. 项目总览 / 决策台
3. 章节写作工作台
4. 研究方向页
5. 论文框架页
6. 文献与引用中心
7. 全文预览 / 审阅 / 定稿页

原因：
- 首页决定第一印象和入口逻辑
- 项目总览决定整个产品的工作流是否顺畅
- 写作工作台决定用户是否愿意长期使用
- 其余页面再逐步补齐，才能形成完整闭环

## 7. 当前执行结论

目前可以明确确认三件事：

1. 旧 UI 不是我们要继续打磨的方向
2. Stitch 已经开始给出更对的学术专业型方向
3. 接下来应当以这份页面蓝图为中心继续生成和落代码，而不是继续围绕旧版结构修补
